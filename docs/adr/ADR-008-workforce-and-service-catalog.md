# ADR-008: Workforce & Service Catalog (Milestone 5)

**Status:** Accepted
**Date:** 2026-07-22
**Milestone:** 5 — Workforce & Service Catalog
**Related:** docs/WORKFORCE_ARCHITECTURE.md, docs/SERVICE_ARCHITECTURE.md, docs/TENANT_ARCHITECTURE.md §4.1, docs/DATABASE_DESIGN.md, docs/PRISMA_SCHEMA.md §5/§14.4, docs/API_SPECIFICATION.md, docs/adr/ADR-006-multi-tenant-saas-engine.md, docs/adr/ADR-007-salon-management.md, docs/IMPLEMENTATION_ROADMAP.md

---

## Decision

Build the Employee/WorkingHours/EmployeeTimeOff domain (`modules/employees`) and the ServiceCategory/Service/EmployeeService domain (`modules/services`) as two new modules, following the same Clean Architecture layering as `modules/salon`/`modules/tenants`. Explicitly **not** built this milestone: Customers, Files/S3 upload, Appointments, Scheduling, WhatsApp, AI, Billing, Analytics — a narrower scope than `docs/IMPLEMENTATION_ROADMAP.md`'s own Milestone 5 (which bundled Employees/Services/Customers/Files together), per the requester's explicit brief. This is the same "narrower/differently-shaped-than-the-roadmap, logged not silent" precedent ADR-003 through ADR-007 already established.

## Context

`docs/IMPLEMENTATION_ROADMAP.md`'s Milestone 5 ("Employees, Services, Customers & Files") was scoped as the full catalog-data milestone: `Employee`, `Service`, `EmployeeService`, plus `Customer`/`CustomerTag`/`CustomerNote` and a `Files`/S3 upload module closing `API_SPECIFICATION.md` §18.2's flagged gap. The requester's brief for this pass is narrower — Employees (profile, status, working hours, time off) and the Service Catalog (categories, services, duration/pricing/buffer, employee eligibility) only, with Customers and Files explicitly deferred to a future milestone. No renumbering is introduced by this ADR (unlike ADR-007's renumbering of the original Milestone 4) — this pass simply narrows Milestone 5's own scope and defers the remainder; see `IMPLEMENTATION_ROADMAP.md`'s own amendment note for the restated boundary.

## Key Design Decisions

| Decision | Summary | Rationale |
|---|---|---|
| Two modules, not one, with a one-directional dependency | `modules/services` (`ServiceCategory`, `Service`) has **no** dependency on `modules/employees`. `modules/employees` (`Employee`, `WorkingHours`, `EmployeeTimeOff`, and the `EmployeeService` junction) imports `ServicesModule` to validate `serviceIds` and read service data. | `SYSTEM_ARCHITECTURE.md`'s module-dependency graph literally lists `Employees → Services` **and** `Services → Employees` — a genuine cycle that would violate the project's own "module graph must be a DAG" rule (§2.4) if built as stated. Resolving it required picking one direction; Employees → Services was chosen because "assign services to this employee" reads as the natural, employee-initiated action matching this milestone's own "Employee ↔ Service assignment" framing, and because `EmployeeService` (the junction needing both sides' data for validation) lives in `modules/employees` as a result. |
| `GET /services/:id` has no hydrated eligible-employees list | The consequence of the decision above: `Service`'s response shape is its own fields only. The reverse view (which employees are eligible for a given service) is available via `GET /employees?filter[serviceId]=`, which already existed in `API_SPECIFICATION.md`'s design. | A `ServicesModule → EmployeesModule` import (needed to hydrate that field) would recreate the cycle. The task's actual requirement — "Employee ↔ Service assignment" / "Employee assignment UI" — is fully satisfied from the employee side; adding the reverse hydration would be scope beyond what was asked, for a field with a working alternative query path. |
| `EmployeeService`'s compound FK is expressible in Prisma's declarative schema — no manual SQL needed | `Employee`/`Service` each carry `@@unique([tenantId, id])`; `EmployeeService.employee`/`.service` use `@relation(fields: [tenantId, employeeId], references: [tenantId, id])`. `prisma migrate dev` generated the compound foreign keys directly — verified by inspecting the generated migration SQL, which shows `ALTER TABLE "employee_services" ADD CONSTRAINT ... FOREIGN KEY ("tenantId", "employeeId") REFERENCES "employees"("tenantId", "id")` with no hand-editing. | `PRISMA_SCHEMA.md` §14.4's table lists the composite-FK cross-tenant pattern as a "required manual migration step" for whichever milestone first needed it. That assumption predates confirming Prisma's actual behavior — modern Prisma supports multi-field relations against a compound unique/`@@id` natively. This ADR corrects that assumption for the record rather than silently doing something different from what the doc said, since a future module modeling its own tenant-owned-to-tenant-owned relation should reuse the declarative approach, not assume a manual-SQL step is required. |
| `EmployeeTimeOff` is a new, separate model — `Holiday` (Milestone 4) is untouched | A new `employee_time_off` table (`employeeId`, `startDate`, `endDate`, `reason?`), owned entirely by `modules/employees`. | `docs/adr/ADR-007-salon-management.md`'s forward note anticipated extending `Holiday` with a nullable `employeeId`, requiring a manual partial-unique-index migration (`WHERE employee_id IS NULL`) to preserve the tenant-wide-duplicate-prevention guarantee. Confirmed with the requester and chosen against: it would touch Milestone 4's already-shipped, tested `holidays` table/migration for a feature (employee leave tracking, a date **range**, not a single closure date) that is conceptually distinct from a salon-wide closure calendar. A dedicated model is simpler, keeps Milestone 4 stable, and doesn't need the tricky partial-index migration at all — `EmployeeTimeOff` has no tenant-wide-duplicate concept to protect against in the first place. |
| `Service.bufferTimeMinutes` — a new, per-service field | Added directly to `Service` (default `0`), distinct from `TenantSettings.business.bookingBufferMinutes` (already reserved, still dormant, for a future tenant-wide buffer the Availability engine will apply between any two bookings). | The task's brief lists "Buffer time" under the Service-domain bullets (alongside duration/pricing), not the tenant-settings bullets — read as a per-service cleanup/processing buffer (e.g., color-treatment processing time), a distinct concept from a tenant-wide minimum-gap policy. Confirmed with the requester. Both fields can coexist and compose once Milestone 6's Availability engine is built (per-service buffer + tenant-wide buffer, summed or maxed — that composition rule is that future milestone's design decision, not this one's). |
| `Employee.userId` — optional link to a login `User`, validated via direct `PrismaService` injection | `EmployeeService` (application layer) reads `prisma.user` directly to validate `userId` belongs to the tenant and isn't already linked, rather than going through a `Users` module. | No dedicated `Users` module exists in this codebase yet (`Auth` owns `User`'s CRUD/session concerns but exports no lookup service) — inventing one solely for this one validation check would be disproportionate scope. This is a narrow, documented exception to the module-boundary convention, not a precedent for reaching into other modules' models freely; a future `Users` module (per `IMPLEMENTATION_ROADMAP.md`'s still-open items) should absorb this check when it exists. |
| `EMPLOYEE_HAS_UPCOMING_APPOINTMENTS` guardrail — not implemented, explicitly deferred | `PATCH`/`DELETE /employees/:id` do not check for conflicting future appointments before deactivating/removing an employee. | No `Appointment` table exists anywhere in the schema yet (Milestone 6 scope). `IMPLEMENTATION_ROADMAP.md`'s own Sprint 5.1 risk note already anticipated this exact situation ("implement the check now against an empty `Appointment` table... add a regression test once Milestone 6 lands") — there being no table at all to implement the check against, it is documented here as an open item for Milestone 6, not silently skipped or stubbed with dead code. |
| Two new RBAC permission keys: `employees:manage`, `services:manage` | Added to `backend/prisma/seed.ts`'s `PERMISSIONS`/`ROLE_PERMISSIONS`, granted to `OWNER`/`MANAGER`. Reads are role-gated only (`STAFF`+), no permission required. | Mirrors the existing `salon:manage` precedent exactly (one permission key per resource domain introduced by a milestone). Two keys (not one shared `workforce:manage`) because Employees and Services are two independently-manageable resource domains with their own controllers/modules — matching the granularity of every other existing permission key. |
| No frontend pagination UI for Employee/Service lists | `GET /employees`/`GET /services` are fetched with a single generous `limit=100` request, matching `AdminApiService.listTenants`'s existing precedent, rather than building pagination controls. | Both are small, per-tenant catalogs (realistically dozens of rows — `API_SPECIFICATION.md` §2.4.2's own reasoning for offering offset pagination on these endpoints in the first place). The one existing offset-paginated list consumer in this codebase (`AdminApiService`) already established this "fetch a generous single page, skip pagination UI" pattern; introducing a new one for a similarly-small dataset would be inconsistent, unrequested scope. |
| Frontend: no signal store, no shared `Modal`/`ServicePicker`/`EmployeePicker` components | Employees/Services pages follow `SalonProfilePage`/`HolidaysPage`'s existing flattened pattern (plain component signals, inline add/edit rows, native checkboxes for service assignment) rather than `FRONTEND_ARCHITECTURE.md`'s fuller aspirational component library. | Restates ADR-007's same precedent: no shipped feature has used the aspirational `Modal`/domain-picker components yet, and this milestone's forms are small enough not to need them. A picker-worthy multi-select would need to actually recur with richer requirements (search, avatars, pagination) before promoting one to `shared/components/`. |

## Consequences

- `docs/IMPLEMENTATION_ROADMAP.md`'s Milestone 5 is amended in place (not renumbered): its scope is narrowed to Employees/Services only for this pass, with Customers and Files explicitly carried forward, restated in that document's own amendment note rather than silently dropped.
- `ServicesModule` is a genuine dependency leaf with respect to `EmployeesModule` — any future module needing both (e.g. Milestone 6's `Availability`/`Appointments`) should import both directly rather than assuming either re-exports the other.
- The composite-FK pattern (`docs/TENANT_ARCHITECTURE.md` §4.1) is now exercised for the first time, and confirmed to require **no manual migration SQL** with the Prisma version this project pins — this corrects `PRISMA_SCHEMA.md` §14.4's original assumption for every future module that introduces its own tenant-owned-to-tenant-owned relation.
- `EmployeeTimeOff` and `Holiday` are two separate, non-unified concepts going forward (employee leave vs. salon-wide closures) — a future Availability engine (Milestone 6) must consult both tables, not assume one supersedes the other.
- `Employee.userId`'s validation living in `EmployeeService` (via direct `PrismaService` access) is a marked, temporary exception — flagged here so a future `Users` module absorbs it rather than this pattern being copied elsewhere by habit.
