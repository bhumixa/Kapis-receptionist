# ADR-005: Authorization / RBAC (Sprint 2.4)

**Status:** Accepted
**Date:** 2026-07-22
**Milestone:** 3 — IMPLEMENTATION_ROADMAP.md calls this slot "Sprint 3.1"; this ADR implements a narrower, RBAC-only charter per the requester's brief — see Context.
**Related:** docs/AUTHENTICATION.md Section 6b, docs/SECURITY.md, docs/API_SPECIFICATION.md Section 2.14, docs/SYSTEM_ARCHITECTURE.md Sections 7.3–7.4/8.4, docs/adr/ADR-002-authentication-schema.md, docs/adr/ADR-003-core-authentication.md, docs/adr/ADR-004-account-security.md

---

## Decision

Implement the RBAC enforcement layer on top of the `Role`/`Permission`/`RolePermission`/`UserRole` schema and `roles: RoleName[]` JWT claim that already existed (ADR-002): a permission resolver with Redis caching, four backend guards (`RolesGuard`, `PermissionGuard`, `TenantScopedGuard`, `SuperAdminGuard`), three decorators (`@Roles`, `@RequirePermission`, `@CurrentTenant`), a request-scoped `TenantContextService`, a guard-layer role-rank interpretation ("role hierarchy"), and an explicit, logged `SUPER_ADMIN` bypass. On the frontend: `PermissionService`, `roleGuard`, an `*appHasPermission` structural directive, and a `filterNavItemsByAccess` navigation-filtering utility.

Explicitly excluded from this sprint: the full `Tenants` business module (`TenantSettings`, `TenantFeature`), the base-repository/composite-FK cross-tenant pattern, the standing tenant-isolation regression suite, the atomic `POST /auth/register` transaction rewrite, `TenantActiveGuard` (frontend or backend), the onboarding wizard shell, and any `Users`/`UserRole` CRUD or role-assignment endpoint. These remain open, roadmap Sprint 3.1 scope.

## Context

`IMPLEMENTATION_ROADMAP.md` calls this slot "Sprint 3.1 — Tenant Context, RBAC Enforcement & Isolation Proof" and bundles it with a full `Tenants` module, a base-repository/composite-FK cross-tenant pattern, and a standing isolation regression suite (see the roadmap's own Sprint 3.1 entry). The requester's brief scoped this sprint narrower — RBAC guards/decorators/permission resolution and tenant-aware authorization checks only — following the same "narrower than the roadmap's original note, logged rather than silently dropped" precedent ADR-003 and ADR-004 already set for the two prior authentication sprints.

What already existed and needed no schema work: `Role`, `Permission`, `RolePermission`, `UserRole` — migrated and seeded (`backend/prisma/seed.ts`) since Sprint 1.1; the JWT access-token payload already carried `roles: RoleName[]` (array, since `UserRole` is many-to-many) and `tenantId: string | null` (`token.service.ts`); `JwtAuthGuard` already set `request.user`, and `AuthModule` already exported `TokenService`/`JwtAuthGuard` specifically for this sprint to compose on (its own doc comment said so); `RedisService` was already in production use (`LoginAttemptService`) via a simple `set(key, val, 'EX', ttl)` pattern. `backend/src/core/` was an empty placeholder directory — this sprint is what fills it. No `Users`/`Tenants` business modules existed, and none are built here.

## Key Design Decisions

| Decision | Summary | Rationale |
|---|---|---|
| Role hierarchy is a guard-layer rank, not a schema change | `ROLE_RANK: Record<RoleName, number>` (`SUPER_ADMIN`=100, `OWNER`=30, `MANAGER`=20, `STAFF`=10) + `satisfiesRoleRequirement()`, duplicated identically on frontend and backend. `@Roles()` is documented to take exactly one role — the minimum required — satisfied by that role or anything ranked higher. | SYSTEM_ARCHITECTURE.md Section 7.3/DATABASE_DESIGN.md describe roles as a flat set with direct `RolePermission` mapping, not a formal inheritance hierarchy. The sprint brief explicitly asks for "role hierarchy" as a deliverable; a rank used only for `@Roles()` satisfaction gives that without touching the data model or the `RolePermission` mapping's directness. |
| Effective permissions are the **union** across all held roles | `PermissionResolverService.getEffectivePermissions(roles)` fans out per-role lookups and unions the results, not a single-role lookup. | `User.roles` is a many-to-many via `UserRole` (ADR-002's own resolution of DATABASE_DESIGN.md's single-`role_id`-FK design), so a user can hold more than one role even though today's seed data never assigns more than one — the resolver has to be correct for that case regardless. |
| Redis cache key is `rbac:role-permissions:{RoleName}`, TTL-only invalidation this sprint | Keyed by the enum string, not a role UUID; default TTL 3600s (`RBAC_PERMISSION_CACHE_TTL_SECONDS`). No active `DEL` on `RolePermission` write. | `RolePermission` rows are only ever written by `prisma/seed.ts` today — no runtime mutation endpoint exists (role-assignment CRUD is explicitly out of scope this sprint). A bounded TTL is sufficient until a real write path exists; DATABASE_DESIGN.md Section 3.1.4 had already flagged Redis caching as the expected design for this hot path. **Active invalidation becomes a real requirement once a runtime role-management endpoint exists — flagged here so it isn't silently forgotten.** |
| `SuperAdminBypassService` is a single shared helper, not inline logic in each guard | Both `RolesGuard` and `PermissionGuard` call `checkAndLog()` first, before touching rank/resolver logic. | Exactly one place grants the bypass and exactly one place logs it — never a risk of one guard bypassing silently while another logs, and it's the single chokepoint a reviewer needs to audit for this deviation's correctness. |
| Bypass logging reuses `SecurityEventService`, not a new mechanism | Added `'SUPER_ADMIN_BYPASS'` to `SecurityEventType`, logged via `.info()` (sanctioned, expected behavior, not a failure/threat event). | Follows the exact "structured log line until a real `AuditLog` table exists" pattern `SecurityEventService` already established (ADR-003/ADR-004) — `AuditLog` remains explicit Milestone 9 scope. |
| `TenantScopedGuard` ships a "resolvable tenant context" baseline only, not per-resource-ID ownership checks | `SUPER_ADMIN` passes unconditionally (no fixed tenant); every other role must carry a non-null `tenantId`. The full check (look up `:id`, compare owning `tenantId`, return `404` not `403` on mismatch) is unbuilt. | No tenant-owned business resource (Employee, Customer, etc.) exists yet to check against — building a speculative pluggable-resolver mechanism against zero real consumers would be premature. `TenantResourceNotFoundException` is reserved in `rbac.exceptions.ts` for whichever future module needs it. |
| `TenantContextService` ships with zero consumers | Request-scoped, exposing `getTenantId()`/`requireTenantId()`/`getCurrentUser()`, per SYSTEM_ARCHITECTURE.md Section 8.3. | No tenant-owned business module or base-repository pattern is in this sprint's scope. Built ahead of its first real user, the same precedent as `TokenService` being built ahead of `RolesGuard` in the prior sprint. |
| A test-only `RbacProbeTestModule`/`RbacProbeController` proves the guards over real HTTP | Mounted only by `test-app.factory.ts`'s `createTestApp()`, never by `src/app.module.ts`. | No production business controller exists yet to attach `@Roles`/`@RequirePermission` to (Users/Tenants CRUD is out of scope). A throwaway production route would be worse than a clearly-labeled, physically-excluded test fixture. |
| `SuperAdminBypassService` is exported from `CoreModule` alongside the guards that depend on it | Discovered empirically: a guard applied via `@UseGuards(GuardClass)` in a module other than the one that declares it needs its own constructor dependencies resolvable from that host module too, not just the guard class itself. | Without this, `RbacProbeTestModule`'s use of `@UseGuards(RolesGuard)` failed DI resolution for `SuperAdminBypassService` even though `RolesGuard` itself was exported and importable. |
| Frontend `ROLE_PERMISSIONS`/`ROLE_RANK` are deliberate, documented duplicates of the backend | No shared types package exists between frontend/backend. | `PermissionService`/`roleGuard` are UX convenience only (FRONTEND_ARCHITECTURE.md Section 5.9) — the backend's `PermissionGuard`/`RolesGuard` remain the sole security boundary regardless of what these maps say. Flagged as an accepted maintenance cost: keep in lockstep with `backend/prisma/seed.ts` and `rbac.constants.ts` whenever a permission key or role's set changes. |
| `*appHasPermission` is hide-only this sprint | No disable variant. | The brief says "hide/disable" — hide is the simpler, sufficient default; a disable variant is a natural, easy follow-on, not built speculatively ahead of a concrete need. |

## The SUPER_ADMIN Bypass — Reasoned Deviation from SYSTEM_ARCHITECTURE.md Section 8.4

SYSTEM_ARCHITECTURE.md Section 8.4 originally specified that `SUPER_ADMIN` should have **zero implicit tenant-scoped power**: *"Super Admin is the sole role that operates without a fixed tenant context, and its guard (`SuperAdminGuard`) is structurally distinct... rather than a 'role that happens to bypass tenant checks' bolted onto the normal path — this avoids a shared code path where a bug could accidentally grant cross-tenant access to a non-admin role."* API_SPECIFICATION.md Section 2.14 restated this: `SUPER_ADMIN` was never meant to be implicitly included in a tenant-scoped endpoint's allowed roles, acting only through the separate `/admin/*` surface.

This sprint's brief explicitly asks for a "Platform Admin bypass" — confirmed with the requester to mean `SUPER_ADMIN` should pass `RolesGuard`/`PermissionGuard` checks on tenant-scoped endpoints directly, not only through a separate admin surface. This is a genuine, deliberate deviation from the original design, accepted with the following mitigations:

1. The bypass lives in exactly one shared, fully unit- and integration-tested chokepoint (`SuperAdminBypassService`), not scattered inline checks — a single place to audit rather than a class of potential bugs spread across every guard.
2. Every bypass use is logged via `SecurityEventService.record('SUPER_ADMIN_BYPASS', { userId, tenantId, route, ...requirement })` — auditable today via structured logs, trivially replayable into a real `AuditLog` table once Milestone 9 builds it, same as every other `SecurityEventService` event.
3. The bypass is scoped narrowly: it only short-circuits **role/permission requirement checks** (`RolesGuard`/`PermissionGuard`). `TenantScopedGuard`'s baseline "resolvable tenant context" check still runs for `SUPER_ADMIN` (it passes unconditionally because it has no fixed tenant by design, not because the check was skipped), and the future per-resource-ID ownership check is explicitly designed to apply to `SUPER_ADMIN` too, unchanged — a Super Admin acting on a specific tenant-owned resource by ID is still expected to go through a resolvable existence/ownership check when that check is eventually built, not silently skip it.
4. `SuperAdminGuard` (the `/admin/*`-surface guard from the original design) is unaffected and still exists, unused this sprint, ready for the future `/admin/*` controllers the original design intended.

## Alternatives Considered

1. **Strict separation, no bypass** — keep `SYSTEM_ARCHITECTURE.md` Section 8.4's original design exactly, treating "Platform Admin bypass" as referring only to the separate `/admin/*` surface. Rejected after confirming with the requester that an in-guard bypass was the intended behavior.
2. **A formal parent-role/hierarchy table in the schema** instead of a guard-layer rank constant. Rejected — the flat `Role`/`Permission`/`RolePermission` schema (ADR-002) stays flat; rank is authorization-layer sugar on top, not a new data model, consistent with DATABASE_DESIGN.md Section 3.1.2's note that custom/tenant-defined roles are a future extension point, not MVP scope.
3. **Active Redis cache invalidation on every `RolePermission` write**, via a signal from `prisma/seed.ts`. Rejected as premature — no runtime write path to `RolePermission` exists yet; a bounded TTL is sufficient until one does.
4. **A throwaway production route** instead of a test-only probe controller, to exercise the guards over real HTTP. Rejected — no production business controller exists yet to attach `@Roles`/`@RequirePermission` to organically, and inventing one purely to prove the guards work would be speculative, unreviewable-for-its-own-sake surface area.
5. **A pluggable per-resource-ID resolver mechanism for `TenantScopedGuard`**, built now ahead of any real consumer. Rejected — no tenant-owned business resource exists yet to design the mechanism against; building it speculatively risks guessing wrong about what the first real consumer (Employees/Customers, Sprint 4.1+) actually needs.

## Consequences

- `backend/src/core/` is no longer empty — `CoreModule` is now part of `app.module.ts`'s import graph.
- `AuthModule` now additionally exports `SecurityEventService` (previously only `TokenService`/`JwtAuthGuard`), and `CoreModule` now exports `SuperAdminBypassService` alongside the guards that depend on it — both required for the DI graph to resolve when guards are applied outside `CoreModule` itself.
- `TenantContextService` ships with zero consumers this sprint — flagged for whichever future module first needs request-scoped tenant context.
- `TenantScopedGuard`'s per-resource-ID ownership check remains a documented open item for Sprint 4.1+ (Employees/Customers), not silently missing.
- API_SPECIFICATION.md Section 2.14's original "`SUPER_ADMIN` is never implicitly included" sentence is now amended, not silently rewritten — see the amendment note in that section.
- The role-change → JWT-invalidation question (a user's already-issued 15-minute access token still carries their old `roles` claim after a role change) remains an open gap — moot in practice this sprint since no code path to change a user's role after registration exists yet (`Users`/`UserRole` CRUD is out of scope), but worth resolving (likely via the existing `SessionService.revokeAllForUser()`) once that CRUD surface is built.
- Frontend `ROLE_PERMISSIONS`/`ROLE_RANK` duplication is a standing maintenance cost: any future change to `backend/prisma/seed.ts`'s permission matrix or `rbac.constants.ts`'s rank values must be mirrored in `frontend/src/app/shared/constants/role-permissions.constant.ts` and `role-rank.util.ts`.
