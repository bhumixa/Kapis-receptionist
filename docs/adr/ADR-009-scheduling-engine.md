## Decision

Build `modules/customers` (deferred out of Milestone 5, picked up now since Appointments needs it), `modules/availability` (pure slot-computation, no controller), and `modules/appointments` (the booking lifecycle) â€” plus two new, reusable cross-cutting primitives: `core/locking/booking-lock.service.ts` (Redis distributed lock) and `core/idempotency/idempotency.interceptor.ts` (generic `Idempotency-Key` handling). Frontend: `features/customers` and `features/appointments` (calendar, booking form, detail/reschedule/cancel).

Explicitly **not** built this milestone: `AppointmentReminder`/any BullMQ reminder worker, `AppointmentFeedback`, `CustomerTag`/`CustomerNote`/`CustomerPreference`, any WhatsApp/AI/Conversation linkage on `Appointment`. Each is named in the requirements as out of scope ("no notifications beyond scheduling") or genuinely blocked on a module that doesn't exist yet (WhatsApp/AI, Milestones 7â€“8) â€” logged here, not silently dropped, the same "narrow the ask, log the deferral" precedent ADR-008 set.

## Context

This is the platform's single most business-critical milestone (IMPLEMENTATION_ROADMAP.md Sprint 6.1: "the highest-complexity single sprint in the roadmap"). Three design questions were resolved with the requester before implementation began, each recorded below because each has real, load-bearing consequences on the schema and API shape actually built.

## Key Design Decisions

| Decision | Summary | Rationale |
|---|---|---|
| **Per-service employee assignment** (confirmed with requester) | A single visit can have different services performed by different employees in sequence, not one employee for the whole appointment. `Appointment.employeeId` is a *denormalized "primary"* value (the first `sequenceOrder` line's employee) for simple `filter[employeeId]` queries and calendar display; `AppointmentService` lines are authoritative for conflict/availability. Each line carries its own computed `[startTime, endTime)` sub-window (sequential, no gap within one visit) plus `blockedUntil`/`isBlocking`. | The simpler "one employee per appointment" alternative was on the table; the requester chose the richer model to match real salon workflows (e.g., color + cut by two different stylists in one visit). This is the single decision with the largest schema/API consequence in the milestone â€” it moved the conflict-prevention unit from `Appointment` to `AppointmentService`, and changed `POST /appointments`' request body shape from the originally-drafted `{ employeeId, serviceIds: [] }` to `{ services: [{ serviceId, employeeId }] }` (API_SPECIFICATION.md Section 10, amended). |

## A Genuine Bug Found and Fixed

The concurrency integration test (see above) surfaced a real, pre-existing race condition in `TenantSettingsService.getSettings()`'s `createDefault` fallback (Milestone 3): `upsert()` alone is not sufficient under true concurrent execution â€” two simultaneous callers can both miss the row on their internal read and both attempt the insert branch, so the second still raises a `P2002` unique-constraint violation on `tenantId` despite being expressed as an upsert. This had never fired before because every tenant created via `POST /auth/register` gets its `TenantSettings` row atomically at registration; Milestone 6's `AvailabilityService.effectiveBufferMinutes` is the first caller to invoke `getSettings` under genuine concurrent load (8 simultaneous booking requests, each computing the buffer). Fixed in `PrismaTenantSettingsRepository.createDefault` by catching the `P2002` and re-reading the row rather than crashing â€” a small, targeted fix to existing Milestone 3 code, verified by the same concurrency test that found it.

## Consequences

- `API_SPECIFICATION.md` Section 10's `POST /appointments`/`.../reschedule` request bodies are amended in place from the originally-drafted single-`employeeId` shape to the per-service-line `{ serviceId, employeeId }` array shape â€” this is a real contract change from the pre-existing draft, not an addition.
- `modules/customers` exists now, one milestone later than `DATABASE_DESIGN.md`'s original Milestone 5 placement â€” `EmployeeService`/`ServiceService`'s existing exported surfaces needed no changes to support it; only `EmployeesModule`, `SalonModule`, and `TenantsModule` gained new `exports` arrays (`WorkingHoursService`/`EmployeeTimeOffService`/`EmployeeAssignmentService`; `BusinessHoursService`/`HolidayService`; `TenantSettingsService`) for their first cross-module consumers.
- `BookingLockService` and `IdempotencyInterceptor` are now standing, reusable primitives â€” a future module needing either should reuse them, not reimplement.
- `AppointmentReminder`/BullMQ reminder scheduling remains fully open for a future milestone; `backend/src/queues/` is still empty.
- The Redis-lock + EXCLUDE-constraint two-layer mechanism, the cursor-pagination utility, and the per-line buffer computation are all reusable precedent for any future high-volume, conflict-sensitive resource.
