# ADR-010: WhatsApp Cloud Platform Integration (Milestone 7)

**Status:** Accepted
**Date:** 2026-07-24
**Milestone:** 7 — WhatsApp Cloud Platform Integration
**Related:** docs/WHATSAPP_ARCHITECTURE.md, docs/MESSAGING_ARCHITECTURE.md, docs/DATABASE_DESIGN.md §2/§3.6-3.7/§10.7, docs/PRISMA_SCHEMA.md §8-9, docs/API_SPECIFICATION.md §11/§2.12, docs/SYSTEM_ARCHITECTURE.md §6, docs/adr/ADR-006-multi-tenant-saas-engine.md, docs/adr/ADR-009-scheduling-engine.md, docs/IMPLEMENTATION_ROADMAP.md

## Decision

Build a single `modules/whatsapp` module (backend) covering `WhatsAppAccount` connection, webhook ingestion, `Conversation`/`Message` persistence, and outbound send — not the three separate modules (`Conversations`, `Messages`, `WhatsApp`) SYSTEM_ARCHITECTURE.md originally sketched. Internally layered per aggregate (own entity/port/service per aggregate), so splitting into separate NestJS modules later, if the module outgrows this shape, is a mechanical refactor, not a design change.

Three genuinely new pieces of cross-cutting infrastructure, none of which had any precedent in this codebase before this milestone:

1. **BullMQ** (`bullmq` + `@nestjs/bullmq`, new dependencies) — `queues/bullmq-root.module.ts` owns a dedicated Redis connection (`maxRetriesPerRequest: null`, required by BullMQ, distinct from the shared `RedisService` connection used for caching/locks/idempotency). Two named queues, `whatsapp-inbound` and `whatsapp-outbound`.
2. **`EncryptionService`** (`core/security/`, AES-256-GCM) — the first *decryptable* secret this codebase stores (`WhatsAppAccount.accessTokenEncrypted`). Every other stored secret (`passwordHash`, `refreshTokenHash`, `tokenHash`) is one-way hashed; this one must be recoverable to make outbound Cloud API calls.
3. **Webhook tenant resolution** — a deliberate, documented second path outside `TenantContextService`. A WhatsApp webhook carries no JWT at all; tenant is resolved by looking up `WhatsAppAccount.whatsappPhoneNumberId` from the payload's `metadata.phone_number_id`, inside the async queue worker, not at the HTTP boundary.

Explicitly **not** built this milestone, each named in the brief's exclusions or the roadmap's own narrower task list, not silently dropped:

- `TemplateMessage` registry (Sprint 7.2's original scope) — a manual reply sent more than 24 hours after the customer's last inbound message is rejected (`422 OUTSIDE_MESSAGING_WINDOW`), not template-routed. The 24-hour customer-service-messaging-window rule itself is enforced.
- Actual media file download/S3 storage — `Message` persists media *metadata only* (Meta's media id, mime type, sha256, filename, size — all present in the inbound webhook payload already, no extra Meta API call needed). The `Files`/S3 module itself remains an open gap carried forward from Milestone 5.
- `ConversationStatus.ESCALATED`/`HUMAN_HANDLING` and any AI hand-off logic — meaningless without an AI auto-responder, which doesn't exist until Milestone 8. `ConversationStatus` is `OPEN`/`RESOLVED`/`CLOSED` only; extending it is that milestone's job.
- AI, billing, analytics, marketing automation — out of scope per the task brief, unrelated to this module's boundaries regardless.

## Context

This is the platform's first real third-party integration and its first background-job consumer. Every prior milestone's Redis usage was synchronous (cache reads, `BookingLockService`'s distributed lock, the idempotency interceptor) — this milestone introduces genuinely asynchronous, retry-driven processing for the first time, and the first stored secret that must be decrypted rather than merely compared.

## Key Design Decisions

| Decision | Summary | Rationale |
|---|---|---|
| Single `whatsapp` module, not three | `WhatsAppAccount`, `Conversation`, `Message`, `WebhookEvent` — one NestJS module, layered per aggregate internally (`domain/{entities,ports}`, `application/*.service.ts` per aggregate, `infrastructure/*.repository.ts`, `interface/*.controller.ts`). | The three-module split in SYSTEM_ARCHITECTURE.md solves an organizational problem (separate ownership) this single-milestone delivery doesn't have yet. Matches the pragmatic narrowing precedent ADR-008/ADR-009 already set repeatedly. Splitting later is mechanical, not a rewrite, since each aggregate already has its own port/service/repository. |
| Two-layer inbound idempotency, mirroring ADR-009's booking-conflict precedent | (1) A fast Redis dedup check (`SET NX EX`, key `dedup:whatsapp:{whatsappMessageId}`, 48h TTL) before any DB write; (2) a partial-unique index on `messages.whatsappMessageId` (`WHERE whatsapp_message_id IS NOT NULL`, manual migration) as the non-bypassable backstop. | Meta's webhook delivery is at-least-once by design (retries on any non-2xx/slow response) — the same "Redis speeds up the common case, the database constraint is the actual correctness guarantee" philosophy ADR-009 established for booking conflicts. Verified with a webhook-replay integration test (identical signed payload sent twice — exactly one `Message` row results) and a simulated cold-Redis race (dedup key absent, DB lookup still catches it). |
| `WebhookEvent` persisted synchronously, before signature verification's outcome is even acted on | The raw payload and a `signatureValid` boolean are written to the database on *every* POST, valid or not, before the fast `200`/`401` response — never only on success. `tenantId` is nullable, resolved later by the async worker. | API_SPECIFICATION.md Section 2.12: a spoofed/invalid-signature request must be forensically visible, not silently dropped, so an attempted attack leaves an audit trail. This also means Meta's own retries of a *transiently failed* valid webhook don't lose data even if the async worker briefly lags. |
| Media stored as plain columns on `Message`, not a separate `Media` model | `mediaWhatsappId`/`mediaMimeType`/`mediaSha256`/`mediaFilename`/`mediaSizeBytes` live directly on `messages`. | The task brief asks for "media metadata persistence," not a media management subsystem — WhatsApp inbound messages carry at most one media object each, and no S3 download happens this milestone (the `Files` module doesn't exist yet either, per its own Milestone 5 deferral). A dedicated `Media` model with its own lifecycle is unwarranted complexity for metadata-only storage; adding one later if real file storage lands is additive, not a migration of existing data. |
| `Message.senderType` reuses the existing `ActorType` enum | No new `MessageSenderType` enum — `senderType: ActorType` (`USER`/`SYSTEM`/`CUSTOMER` populated this milestone, `AI` reserved for Milestone 8). | The concepts are identical (who/what originated an action), and `ActorType` is already the platform-wide convention for this (`AuditLog.actorType`, `AppointmentStatusHistory.actorType`). Introducing a parallel enum for the same concept would be pure duplication. |
| `ConversationStatus` narrowed to `OPEN`/`RESOLVED`/`CLOSED` | No `OPEN_AI`/`ESCALATED`/`HUMAN_HANDLING` this milestone. | Those states describe AI-to-human hand-off, which requires an AI actor to hand off *from* — Milestone 8's job. Every conversation this milestone is human-monitored by default (`OPEN`); extending the enum when AI lands is additive (new enum values, no data migration), not a redesign. |
| Webhook tenant resolution bypasses `TenantContextService` entirely | `InboundMessageProcessorService` resolves tenant via `WhatsAppAccountRepositoryPort.findByPhoneNumberId(metadata.phone_number_id)` inside the BullMQ worker — a global (non-tenant-scoped) lookup, the only such lookup in this codebase outside `Admin`/webhook contexts. | Every other request path in this system assumes an authenticated JWT carrying a `tenantId` claim; a WhatsApp webhook has no JWT at all. This is a genuine, necessary second tenant-resolution mechanism, not a bug — documented here explicitly per ADR-006's own note that `TenantContextService` being the sole resolver has this one structural exception. |
| `EncryptionService` is generic (`core/security/`), not WhatsApp-specific | AES-256-GCM encrypt/decrypt, a new required env var `WHATSAPP_TOKEN_ENCRYPTION_KEY` (32 bytes, base64), validated at bootstrap (fail-fast, matching `JWT_ACCESS_SECRET`/`JWT_REFRESH_SECRET`'s existing precedent). | `WhatsAppAccount.accessTokenEncrypted` is this codebase's first *decryptable* stored secret (everything else is one-way hashed). Naming and placing the service generically (not `WhatsAppTokenEncryptionService`) means a future Stripe-key or similar decryptable-secret need (Milestone 9, Billing) can reuse it directly. |
| BullMQ on a dedicated Redis connection, not the shared `RedisService` | `queues/bullmq-root.module.ts` opens its own `ioredis` connection with `maxRetriesPerRequest: null`. | BullMQ requires that setting on any connection it drives (its blocking-command retry semantics conflict with a finite `maxRetriesPerRequest`), which would be wrong for `RedisService`'s other, latency-sensitive callers (`BookingLockService`, `IdempotencyInterceptor`). |
| Outbound retry: BullMQ backoff for transient failures, terminal for permanent ones | `WhatsAppCloudApiError.isTransient` (5xx/429) → rethrown, BullMQ retries with exponential backoff (5 attempts). Non-transient (4xx) → caught, `Message.status` set `FAILED` immediately, audit-logged, never retried. A `failed` worker-event listener also marks `FAILED` once BullMQ's own retry budget is exhausted. | Matches SYSTEM_ARCHITECTURE.md Section 6.5's documented retry-classification rule exactly — retrying an invalid-recipient or revoked-token error indefinitely wastes attempts and delays staff visibility into a real problem. |
| New RBAC permission: `whatsapp:manage` | Granted to `OWNER`/`MANAGER` only, gating `POST`/`DELETE /whatsapp/account` (connecting/disconnecting the integration). Conversation/message read and reply (`GET /conversations`, `GET/POST /messages*`) are `STAFF`-broad, matching the existing `appointments:manage`/`customers:manage` precedent of gating only the most sensitive action per resource. | Connecting an external messaging credential is comparable in sensitivity to billing/account-level actions; replying to a customer conversation is normal front-desk work, same as creating an appointment or customer record. |

## Consequences

- `bullmq`/`@nestjs/bullmq` are now standing dependencies; `src/queues/` is no longer an empty placeholder. Any future job-queue need (e.g., Milestone 9's dunning notifications, Milestone 6's still-deferred appointment reminders) should reuse `BullmqRootModule`'s shared connection rather than opening a second one.
- `EncryptionService`/`EncryptionModule` (`core/security/`) is a new, reusable cross-cutting primitive alongside `AuditLogService`/`IdempotencyInterceptor`/`BookingLockService`.
- `CustomerService` gained one new public method, `findOrCreateByPhoneForTenant` — the contact-synchronization entry point inbound WhatsApp messages use; it does not otherwise change Milestone 6's Customers module.
- `main.ts`/`test-app.factory.ts` both now boot with `rawBody: true` (required for `X-Hub-Signature-256` verification against the exact byte sequence Meta signed) and exclude `webhooks/whatsapp` from the `/api/v1` prefix and the success-response envelope, the same treatment `/health` already has.
- `docs/DATABASE_DESIGN.md`/`docs/PRISMA_SCHEMA.md`'s original WhatsApp/Conversation design is narrowed as documented above; this ADR is the record of the as-built deviation.
- Live inbound webhook delivery requires a public HTTPS callback URL registered in Meta's App Dashboard — an operational step outside this codebase, tracked as an external dependency the same way Sprint 1.1's WhatsApp Business verification already was.
