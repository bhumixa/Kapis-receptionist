# ADR-007: Salon Management (Milestone 4)

**Status:** Accepted
**Date:** 2026-07-22
**Milestone:** 4 — Salon Management
**Related:** docs/SALON_ARCHITECTURE.md, docs/TENANT_ARCHITECTURE.md §4.1, docs/DATABASE_DESIGN.md, docs/PRISMA_SCHEMA.md §4/§14.4, docs/API_SPECIFICATION.md, docs/adr/ADR-006-multi-tenant-saas-engine.md, docs/IMPLEMENTATION_ROADMAP.md

---

## Decision

Build the salon business-profile domain only — salon profile (business/contact info, description), branding (logo placeholder, colors), currency, weekly business hours, and holiday management — as a new `modules/salon` following the same Clean Architecture layering as `modules/tenants`. Explicitly **not** built this milestone: Employees, Services, Customers, Files/S3 upload, Scheduling, WhatsApp, AI, Billing, Analytics — per the requester's explicit scope boundary, which narrows and re-orders the project's own `docs/IMPLEMENTATION_ROADMAP.md` (that document previously bundled Employees/Services/Customers/Files into "Milestone 4"; this ADR's scope is deliberately different, and the roadmap has been renumbered accordingly — see that document's own change log).

## Context

`docs/IMPLEMENTATION_ROADMAP.md`'s original Milestone 4 ("Salon Management") was scoped as the catalog data every later milestone depends on — Employees, Services, `EmployeeService`, `BusinessHours`, and a `Files`/S3 upload module, closing out `docs/API_SPECIFICATION.md` §18.2's flagged gaps. The requester's brief for this milestone is narrower and differently shaped: salon profile/branding/business-hours/holidays only, with Employees/Services/Customers/Files explicitly deferred to a later milestone (now renumbered Milestone 5). This is the same "narrower/different-than-roadmap, logged not silent" precedent ADR-003/004/005/006 already established for prior milestones.

## Key Design Decisions

| Decision | Summary | Rationale |
|---|---|---|
| `Tenant` already IS the salon; no new `Salon` root entity | `SalonProfile` is a 1:1 satellite table holding only new fields (description, contact, currency, branding) — `Tenant`'s existing `name`/`timezone`/`defaultLocale`/`addressLine1/2`/`city`/`countryCode` are reused via `TenantService`, never duplicated. | This is a single-location MVP (`docs/PROJECT_REQUIREMENTS.md`'s multi-location support is explicitly future-scope); `Tenant` already models the salon's core identity since Milestone 2/3. Introducing a parallel `Salon` table with its own `name`/`timezone` would create two sources of truth for the same facts and contradict the already-approved `Tenant`-is-the-tenant-root architecture. |
| `SalonProfile` mirrors `TenantSettings`' 1:1-satellite-table precedent, not flat columns on `Tenant` | New table, `tenantId @unique` FK, `GET /salon` auto-vivifies a default row on first read exactly like `TenantSettingsService.getSettings`. | Reuses an established, tested pattern rather than inventing a new one; keeps `modules/tenants`' own schema/migration history untouched by this milestone's additions. |
| `PATCH /salon` composes two module's writes in one transaction, via one small, backward-compatible change to `modules/tenants` | `TenantRepositoryPort.updateProfile` and `AuditLogService.record` both gained an optional trailing `tx?: Prisma.TransactionClient` parameter (every existing call site omits it, unaffected). `SalonProfileService.updateProfile` opens one `prisma.$transaction`, calling `TenantService.updateProfile(..., tx)` for the Tenant-owned subset and its own repository's `upsert(..., tx)` for the SalonProfile-owned subset. | The alternative — writing to `prisma.tenant` directly from `modules/salon` — would violate the module-boundary rule (`SYSTEM_ARCHITECTURE.md` §2.3: "no module reaches into another module's Prisma models directly") established since Milestone 3. A second alternative — two sequential, non-transactional writes — was rejected: a mid-request failure could leave a Tenant-field update applied without its paired SalonProfile update (or vice versa), an inconsistency worth the small, explicit `tx`-threading cost to prevent. This is the one deliberate, reviewed touch to Milestone 3 code this milestone makes. |
| `BusinessHours`/`Holiday` field names/shapes match `docs/PRISMA_SCHEMA.md`'s already-reserved, broader (Employee/Branch-aware) design | Built as the exact subset that scope supports — no `employeeId`/`branchId` columns, since Employees/Branch don't exist yet — rather than inventing `Salon`-prefixed model names. | `PRISMA_SCHEMA.md` §4 already designed these two models in full (for a later milestone), anticipating this exact scope boundary. Reusing the planned names/fields means a future milestone's migration only *adds* columns to these tables; inventing differently-named tables now would create a rename/merge problem the moment Employees/Branch land. |
| `logoUrl` lives on `SalonProfile` as a bare placeholder string, not on `Tenant` | `modules/auth`'s `TenantResponseDto.logoUrl` (dormant since Milestone 3, always `null`) is left untouched except for a one-line comment; the real placeholder field lives on the new `SalonProfile` table instead. | `PRISMA_SCHEMA.md` already reserves `Tenant.logoFileId` as a real, S3-backed relation for the eventual `Files` module. Reusing the `logoUrl` name/shape on `Tenant` now — even as a placeholder — would collide with that already-approved future design and create migration friction later. A placeholder string genuinely belongs on the new profile table, not the tenant root. |
| `Holiday.reason` is required (not nullable) | Deviates from `PRISMA_SCHEMA.md`'s eventual nullable `reason` (designed to accommodate a future per-employee ad hoc day-off entry with no label). | This milestone's holidays are always salon-wide and calendar-facing — a display label is always meaningful and expected by the UI; a required field is the correct constraint for *this* milestone's actual use case, without blocking a future migration from relaxing it if a genuine no-label use case appears. |
| `Holiday.isRecurringAnnually` — considered, rejected | A "repeats every year" flag was discussed and explicitly not built. | Would require an availability engine to "expand" a recurring holiday into concrete dates — real semantic complexity disproportionate to a salon-profile-only milestone with no Availability engine yet (that's a future milestone). Noted here as a deliberate, reasoned rejection, not an oversight, so it isn't silently re-proposed without this context. |
| `Holiday` uses a plain `@@unique([tenantId, date])`, not a partial index | Sufficient because this milestone has no `employeeId` column at all. | `docs/DATABASE_DESIGN.md` already anticipates that the eventual `(tenant_id, date, employee_id)` constraint must be a **partial** unique index (`WHERE employee_id IS NULL`), since Postgres treats every `NULL` as distinct in an ordinary composite unique index — a plain 3-column `@@unique` would not prevent duplicate tenant-wide holidays once `employeeId` is nullable. Flagged in `docs/SALON_ARCHITECTURE.md` as a forward requirement for whichever milestone adds `employeeId`, not something to naively copy from this milestone's simpler constraint. |
| Composite-FK cross-tenant pattern remains dormant | No manual post-migration SQL needed — `SalonProfile`/`BusinessHours`/`Holiday` all FK to `Tenant` directly, none reference another tenant-owned entity. | Consistent with `docs/TENANT_ARCHITECTURE.md` §4.1's original framing: the pattern becomes load-bearing only once a real tenant-owned-to-tenant-owned relation exists (e.g. a future `Employee`↔`Service`). This milestone doesn't introduce one. |
| No `Files`/S3 upload — logo is a plain URL field | `POST /files` (pre-signed upload) explicitly deferred to whichever future milestone builds `Files`. | Matches the requester's explicit instruction ("Logo upload placeholder (storage integration later)"); building real upload infrastructure now would be scope creep against an explicit boundary. |
| New RBAC permission `salon:manage`, granted to `OWNER`/`MANAGER` | Added to `backend/prisma/seed.ts`'s `PERMISSIONS`/`ROLE_PERMISSIONS`. Reads are role-gated only (`STAFF`+), no permission required — mirrors the existing `GET/PATCH /tenant` read/write split. | Every guarded mutation needs its own permission key per this codebase's seed-file convention ("each milestone that introduces a guarded action is responsible for adding its permission key(s) here"); no existing key fit this new resource. |
| Frontend: no signal store, no shared `Modal` component built | Three pages follow `SettingsPage`'s existing flattened pattern (plain component signals); holidays use inline add/edit forms instead of a modal dialog. | `FRONTEND_ARCHITECTURE.md`'s fuller `pages/components/services/state/models` structure and `shared/components/primitives/Modal` are aspirational — no shipped feature has used either yet. Matching the one real, working precedent (`SettingsPage`) is more consistent than introducing new shared infrastructure for a single consumer; a modal-worthy form would need to actually recur before promoting one to `shared/components/primitives/`. |

## Consequences

- `docs/IMPLEMENTATION_ROADMAP.md` is re-scoped: Milestone 4 becomes salon-profile-only (this ADR's scope); the original Milestone 4 content (Employees/Services/Customers/Files) moves to a new Milestone 5, cascading every subsequent milestone's number up by one (old 5→6, 6→7, 7→8, 8→9, 9→10, 10→11). `docs/TENANT_ARCHITECTURE.md` §4.1, `docs/SECURITY.md`, and ADR-002/ADR-006 each carry a short amendment note pointing here rather than being rewritten (they're historical records of what was true at the time).
- `TenantRepositoryPort.updateProfile` and `AuditLogService.record` both permanently carry an optional `tx` parameter now — any future module composing a transactional write with `TenantService`/`AuditLogService` can reuse this same pattern rather than re-deriving it.
- A `PATCH /salon` that touches both a Tenant field and a SalonProfile field writes two audit rows (`SALON_PROFILE_UPDATED` and `TENANT_PROFILE_UPDATED`) — an accepted, documented consequence of reusing `TenantService` rather than duplicating its update logic (see `docs/SALON_ARCHITECTURE.md` §3).
- `ApiClient` (frontend) gained a `put<T>()` method, its first real consumer being `PUT /salon/business-hours` — every future full-replace endpoint can reuse it without re-adding the method.
- The `holidays`/`business_hours` Prisma models now exist with the exact field names/shapes a future Employees/Branch milestone expects — that milestone's migration is purely additive (new columns, a replaced unique index for `Holiday`) rather than a rename.
